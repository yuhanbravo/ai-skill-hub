# ChatGPT Handoff Pilot

使用这个 skill 作为最小母版，帮助真实项目建立一种稳定协作方式：

- ChatGPT 负责架构、方案拆解、边界说明、审稿
- Codex 或 Copilot 负责读取任务包、按范围实施、输出执行回执

本 skill 只定义最小流程和边界约束，不绑定任何仓库结构，也不内置自动化脚本。

## 适用场景

- 已经有上游任务包，需要实施侧严格照单执行
- 希望把“方案制定”和“代码落地”分离
- 需要在真实项目中先做小范围试跑，再逐步补充规范
- 希望执行回执里明确记录改动、风险和假设

## 标准 Workflow

1. 读取任务包。
   先完整阅读上游提供的任务包、约束、验收标准和目标文件范围；若任务包引用了 `AGENTS.md`、`ai/tasks/` 或其他项目约定，也一并纳入上下文。
2. 先总结理解，再开始实施。
   在真正动手前，用简短文字复述本次目标、边界、计划修改范围和不做事项，确保上游与实施侧对任务理解一致。
   复述建议控制在 2-4 句话，并明确包含：
   - 本轮目标是什么
   - 本轮明确不做什么
   - 本轮预期输出文件是什么
3. 按范围实施。
   仅在任务包明确授权的范围内修改文件、补充文档或实现代码；优先完成直接目标，不顺手扩展需求，不做无关整理。
4. 输出 execution report。
   完成后输出一份简洁回执，至少说明：实际改动内容、是否完成目标、未完成项、风险点、测试或验证情况、需要上游确认的事项。

## 任务包最小必备字段

上游任务包建议至少包含以下字段：

- 任务名称或任务包标识
- 背景与目标
- 本次范围
- 明确不做什么
- 目标文件或目标目录
- 验收标准
- 约束条件
- 输出要求

如果缺少上述关键字段，实施侧应先用最小假设补齐理解，并在 execution report 中显式记录。

## 范围控制

- 严格控制范围，不做无关重构
- 不主动重命名模块、目录或接口，除非任务包明确要求
- 不顺手修复与当前任务无直接关系的问题
- 如果发现额外问题，只在 execution report 中记录，不混入本次实施

## 额外问题处理

- 如果发现与当前任务直接冲突的问题，应先缩小改动面，优先保证本次目标可交付
- 如果发现可顺手修复但未被授权的问题，默认不修，只做记录
- 如果发现会影响验收的阻塞项，应在 execution report 中单列“阻塞/待确认项”
- 如果项目本地协作规则要求先回报再施工，应遵守项目本地规则

## 假设处理

- 如果任务包存在空白、歧义或缺少上下文，可以做最小必要假设后继续
- 所有假设都必须显式写入 execution report
- 假设应尽量具体，方便上游快速确认或修正

## Execution Report 最小建议结构

- 任务名称或任务包标识
- 对任务理解的简短复述
- 本次实际修改
- 未修改但已确认的边界
- 验证情况
- 风险与待确认项
- 假设清单

execution report 应聚焦本次实施本身，不扩写成长篇方案，不替代项目正式设计文档。

## 使用提醒

- 这是母版 skill，不代表已经完成真实项目接入
- 真正落地时，应复制到目标项目的 `.codex/skills/` 中，再结合项目内约定试跑
- 若项目已有自己的交接文档、任务目录或代理规范，应以项目本地规则为准
- 若本 skill 与项目本地规则冲突，以项目本地规则为准；本 skill 只提供最小兜底流程
