# <Skill Name>

## 1. 背景（Problem Context）

这个 skill 用来解决什么问题。

它应该回答：

- 当前痛点是什么
- 为什么需要这个 skill
- 它在整个 skill-hub 或项目协作中的位置是什么

建议写法：

- 用中文说明真实问题背景
- 不要直接堆命令或参数
- 先说明“为什么存在”，再说明“做什么”

## 2. 适用场景（When to Use）

适合在以下场景使用：

- <场景 1>
- <场景 2>
- <场景 3>

建议写法：

- 用“什么时候应该调用这个 skill”来写
- 优先描述任务场景，而不是功能列表

## 3. 不适用场景（When NOT to Use）

以下情况不适合使用这个 skill：

- <不适用场景 1>
- <不适用场景 2>
- <不适用场景 3>

建议写法：

- 明确边界，避免误用
- 如果项目已有更成熟的本地机制，应说明以本地规则为准

## 4. 核心模式（Pattern）

### <Pattern Name>（<中文模式名>）

这个 pattern 的核心是：

- <核心对象 / 核心动作 1>
- <核心对象 / 核心动作 2>
- <核心对象 / 核心动作 3>

建议进一步说明：

1. 输入是什么
2. 中间动作是什么
3. 输出是什么
4. 为什么这样组织

建议写法：

- pattern 名使用“英文 + 中文”
- 这里要写方法论，不要只写命令行
- 如果是 tool 型 skill，也要抽象出“如何使用这个工具”的稳定模式

## 5. 核心原则（Principles）

- <原则 1，中文主表达>  
  <Principle 1 in English>

- <原则 2，中文主表达>  
  <Principle 2 in English>

- <原则 3，中文主表达>  
  <Principle 3 in English>

建议写法：

- 原则使用“中文 + 英文锚点”
- 数量建议 3-6 条
- 写成稳定原则，不要写成一次性步骤

## 6. 执行流程（Execution Steps）

1. <步骤 1>
   说明这一阶段要读取什么、判断什么、产出什么。

2. <步骤 2>
   说明这一阶段如何继续推进。

3. <步骤 3>
   说明执行中的范围控制、判断标准或分支处理。

4. <步骤 4>
   说明最终输出或回执。

建议写法：

- 这是 AI 最直接会照着执行的部分
- 写成“可执行步骤”，不要写成纯概念说明
- 如果 skill 是 project 型，优先体现阶段感
- 如果 skill 是 tool 型，优先体现输入、处理、输出链路

## 7. 约束（Constraints）

- <约束 1>
- <约束 2>
- <约束 3>

建议写法：

- 写“不能做什么”“默认不做什么”“优先遵守什么”
- 约束优先写硬边界，而不是建议

## 8. 风险（Risks）

- <风险 1>
- <风险 2>
- <风险 3>

建议写法：

- 风险不是泛泛而谈，要尽量贴近这个 skill 的真实误用方式
- 可以写“输入不完整的风险”“边界不清的风险”“输出误导的风险”

用 1-2 段总结这个 skill 的定位：

- 使用时应优先遵守什么规则

## 使用说明（How to Use This Template）

这个模板适用于 `pattern`、`tool`、`project`、`mixed` 四类 skill，但可以按类型微调重心：

- `pattern` 型：重点强化 `Pattern`、`Principles`、`Reusable Prompt`
- `tool` 型：重点强化 `Execution Steps`、`Constraints`、`Risks`
- `project` 型：重点强化 `Problem Context`、`Execution Steps`、阶段划分
- `mixed` 型：先明确主类型，再决定哪一部分写得更重

## 语言规范（Language Rules）

- 正文：中文
- Pattern 名：英文 + 中文
- Principles：中文主表达 + 英文锚点
- 技术元素：英文
- 命令、路径、文件名、参数名：使用代码格式

## 最低验收标准（Minimum Acceptance Checklist）

- `README.md` 和 `SKILL.md` 职责分离
- `SKILL.md` 能让 AI 直接理解怎么执行
- `Pattern` 不是空标题，必须有可复用抽象
- `Principles` 不是步骤改写，必须是稳定原则
- `Execution Steps` 不是纯介绍，必须有执行顺序
- `Constraints` 和 `Risks` 不能缺失
- `Reusable Prompt` 至少有一个可直接复制使用的模板
