# ChatGPT Handoff Pilot

## 1. 背景（Problem Context）

这个 skill 用来建立一种最小可用的 handoff 协作方式，让上游负责方案与边界，下游负责按任务包实施，并在完成后回传 execution report。

它解决的核心问题不是“如何写更多规则”，而是“如何在真实项目里先把交接输入、实施边界和执行回执稳定下来”。因此它刻意保持为母版，不绑定固定仓库结构，也不内置自动化脚本。

## 2. 适用场景（When to Use）

- 已经有上游负责人，需要先产出任务包，再交给实施侧代理执行
- 希望把“方案制定”和“代码落地”明确分开
- 希望实施侧严格按范围施工，避免顺手扩展
- 希望每轮执行后都有统一的 execution report
- 希望先在真实项目中小范围试跑 handoff 流程，再逐步补充规范

## 3. 不适用场景（When NOT to Use）

- 任务本身很小，直接由同一个人或同一个代理完成更高效
- 项目已经有更成熟且强制执行的本地 handoff 规范
- 上游无法提供最基本的任务目标、边界或验收要求
- 当前工作需要开放式探索、快速试错，而不是边界清晰的交付执行

## 4. 核心模式（Pattern）

### Task Package Handoff Pattern（任务包交接模式）

这个 pattern 的核心是把一次协作稳定拆成三个对象：

1. `task package`
   上游输出的任务包，负责定义目标、范围、约束、验收标准和预期交付物。
2. `bounded execution`
   实施侧只在授权范围内落地，不顺手扩展，不把额外问题混入本轮交付。
3. `execution report`
   实施完成后的回执，负责记录实际改动、边界外内容、验证情况、风险和假设。

这不是为了增加流程感，而是为了降低多代理协作中的理解偏差、范围漂移和回执缺失。

## 5. 核心原则（Principles）

- 明确输入，先于实施。  
  Clarify the input before execution.

- 边界优先，避免扩写。  
  Keep the execution bounded and avoid scope creep.

- 先复述理解，再开始改动。  
  Restate the understanding before making changes.

- 假设必须显式记录。  
  Every assumption must be made explicit.

- 回执聚焦结果，不重写方案。  
  The execution report should focus on outcomes, not restate the whole plan.

- 以项目本地规则为准。  
  Defer to repository-local rules when they already exist.

## 6. 执行流程（Execution Steps）

1. 读取任务包。  
   完整阅读上游提供的任务包、约束、验收标准和目标文件范围；若任务包引用了 `AGENTS.md`、`ai/tasks/`、`docs/HANDOFF.md` 或其他项目约定，也一并纳入上下文。

2. 检查任务包是否具备最小字段。  
   建议至少具备以下字段：
   - 任务名称或任务包标识
   - 背景与目标
   - 本次范围
   - 明确不做什么
   - 目标文件或目标目录
   - 验收标准
   - 约束条件
   - 输出要求

3. 先复述理解，再开始实施。  
   在真正动手前，用 2-4 句话总结：
   - 本轮目标是什么
   - 本轮明确不做什么
   - 本轮计划修改哪些文件或目录
   - 本轮预期输出什么

4. 按范围实施。  
   仅在任务包明确授权的范围内修改文件、补充文档或实现代码；优先完成直接目标，不顺手扩展需求，不做无关整理。

5. 处理额外问题。  
   如果发现额外问题：
   - 与当前任务直接冲突：优先缩小改动面，保证本轮目标可交付
   - 可顺手修复但未被授权：默认不修，只做记录
   - 会影响验收的阻塞项：在回执中单列“阻塞/待确认项”

6. 输出 execution report。  
   完成后输出一份简洁回执，至少说明：
   - 任务名称或任务包标识
   - 对任务理解的简短复述
   - 本次实际修改
   - 未修改但已确认的边界
   - 验证情况
   - 风险与待确认项
   - 假设清单

## 7. 约束（Constraints）

- 这是最小母版 skill，不绑定任何固定业务项目结构
- 不强制要求固定文件名，只约定“任务包 / 执行 / 回执”这三类对象
- 不内置脚本，不提供自动化生成器
- 不替代项目本地的正式设计文档、交接文档或协作规范
- 如果项目已有成熟目录结构，应优先沿用项目本地目录

## 8. 风险（Risks）

- 如果任务包字段不完整，实施侧容易做出错误假设
- 如果边界说明不清晰，实施侧容易出现 scope creep
- 如果 execution report 过于笼统，上游难以判断是否真正完成
- 如果本 skill 被误当成“完整框架”，可能导致过早过度设计
- 如果与项目本地规则冲突却未声明优先级，协作会出现双重标准

## 9. Prompt 模板（Reusable Prompt）

### 模板 1：上游给 ChatGPT，用来产出任务包

```text
请使用 `chatgpt-handoff-pilot` 的协作模式，生成一份可以交给 Codex / Copilot 执行的任务包。

背景：
- <背景>

目标：
- <目标>

本次范围：
- <范围>

明确不做：
- <不做事项>

目标文件或目录：
- <文件或目录>

验收标准：
- <验收标准>

约束条件：
- <约束>

输出要求：
- 请输出结构清晰、边界明确的任务包
- 如果上下文不完整，请补充“假设前提”
- 任务包应能直接用于 handoff
```

### 模板 2：上游直接给 Codex / Copilot，用来执行任务包

```text
请按 `chatgpt-handoff-pilot` 模式执行下面的任务包。

要求：
- 先完整阅读任务包
- 动手前先用 2-4 句话复述目标、边界、修改范围和不做事项
- 只在授权范围内实施
- 完成后输出 execution report

任务包：
<粘贴任务包正文>
```

### 模板 3：实施完成后的 execution report 提示词

```text
请基于本次实施结果输出 execution report。

请按以下结构组织：
- 任务名称或任务包标识
- 对任务理解的简短复述
- 本次实际修改
- 未修改但已确认的边界
- 验证情况
- 风险与待确认项
- 假设清单

附加要求：
- 有阻塞项则单列说明
- 有未完成项则说明原因
- 额外发现的问题只记录，不混入本次完成项
```

## 10. 总结（Summary）

`chatgpt-handoff-pilot` 的目标不是把 handoff 做复杂，而是把一次交接最关键的输入、边界和回执稳定下来。

如果真实项目已经有更强的本地协作规则，应优先遵守本地规则；这个 skill 提供的是一个最小可复用的 handoff pattern，用于快速建立可执行、可复盘、可迁移的协作基线。
